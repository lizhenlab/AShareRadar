# Maintenance and Refactor Guide

## 1. Engineering Rules

- Keep HTTP routes thin. Route files should validate inputs and call workflows/services.
- Keep provider logic behind `DataHub` and `provider_registry`.
- Keep SQLite access in repositories or `SQLiteCache`; do not write SQL in route handlers.
- Keep provider routes behind `run_api()`. Async handlers that call synchronous repositories, scheduler state, or diagnostics must use `run_sync_api_async()`; reserve `run_sync_api()` for synchronous boundaries. Client `RequestValidationError` stays `422`, while internal Pydantic `ValidationError` must be logged and mapped to generic sanitized `503` detail.
- Preserve `SameOriginMutationMiddleware` ahead of route side effects for `POST`/`PUT`/`PATCH`/`DELETE` and truthy `GET ...?refresh=...`. Browser-origin requests must separately validate the Host-derived request origin and supplied Origin/Referer against configured origins so a matching hostile Host cannot bypass the boundary through Host/DNS rebinding. Keep default-port normalization, metadata-only cross-site rejection, ordinary read-only access, and non-browser clients without origin metadata covered when changing middleware or CORS settings.
- Keep user input/update Pydantic models strict about unknown fields unless a compatibility requirement explicitly says otherwise.
- Keep every committed user-data import behind `exclusive_local_data_operation()`, a fresh single-use preview claim bound to path/mode/bundle/current user state, and a verified rollback backup created before mutation. Do not move claim consumption, backup, or import outside that serialized boundary.
- Keep browser file reads, import previews/commits, and cleanup previews under explicit latest-request ownership. A stale completion must not revoke or repaint a newer selection, token, feedback message, or action state. Only a successful current import may invalidate browser caches, reload runtime-owned panels, and rebuild the SSE subscription.
- Keep scheduled cleanup on `cleanup_regenerable_runtime_rows()`. User-history retention belongs only to the explicit previewed cleanup flow, which must back up before deletion and preserve advice referenced by review plans.
- Keep current-day review and watchlist-scan requests free of a synthetic end-of-day `as_of`; let the server use current Shanghai time. Past dates use Shanghai-local end of day, and review actions remain bound to the symbol that owns the rendered panel during a pending stock switch.
- After an alert write succeeds, reconcile the affected rule locally before any readback. A failed readback must report synchronization degradation without restoring stale row content or claiming the write failed.
- Add or update tests before changing trend, alert, provider fallback, or data-quality behavior.
- Do not store API keys in project files.
- Keep the app runtime isolated from user-level Python packages; optional provider stacks such as pandas/numpy must resolve from the project runtime, not `~/.local`.
- Keep direct runtime libraries in `requirements.txt` and direct engineering tools in `requirements-dev.txt`. Rebuild both Python 3.12 hashed locks when the runtime input changes, and rebuild the development lock when only the development input changes. Reproducible installs must use `--require-hashes` against a lock file.
- Keep one Uvicorn worker as the supported scheduler topology. Preserve the `<SQLite path>.scheduler.lock` cross-process guard, bounded shutdown, manual-run ownership, and cancellation-dominant task history. A stop timeout must defer lock release until unfinished runners exit; `cancelled` may replace a racing `success`, but late `success`/`failed` updates must not replace `cancelled`.
- Keep tests hermetic: use the active project runtime with `PYTHONNOUSERSITE=1`, temporary SQLite paths, fake providers/clients, and no required live network calls.
- When adding a user-facing recommendation, include reason, risk, invalidation condition, or missing-data note.
- When research logic consumes prices, volumes, forward returns, confidence, or factor scores from provider/cache/calibration inputs, guard non-finite values before text or statistics are assembled.
- Treat feature snapshots as a shared research boundary: clean non-finite or non-positive prices, amount, turnover, volume ratio, ATR/volatility, MA/support/resistance, and research score fields before leadership, Alpha, factor, or diagnosis text consumes them.
- Rule matches, minute-analysis plans, and market-regime evidence must turn invalid numbers, insufficient samples, and blank names into missing-data/fallback text instead of actionable or empty evidence. Minute rows must be timestamp-sorted/de-duplicated and restricted to the latest Shanghai-local market date; unavailable reports may retain filtered rows only as audit data and must clear executable ranges.
- Treat persistence/logging after a successful provider response as a local side effect. Cache writes, capability-status writes, runtime-event writes, and workbench local-state reads should degrade visibly or log best-effort instead of converting usable provider data into a loading failure. Market-sampling logs, quote-confirmation/profile persistence, and quote-stream watchlist fallback reads must use the SQLite offload helpers rather than running on the event-loop thread.
- Keep synchronous SQLite work off async event-loop threads through `run_sync_api_async()`, `run_cache_io()`, or scheduler offload helpers. Blocking provider SDK calls belong in the `ProviderRuntime`-owned bounded executor, and every DataHub lifecycle path must reach idempotent `DataHub.aclose()`/provider close cleanup.
- Sanitize provider errors at both persistence directions: before request/status write output and again in status row mappers. Tests must cover URL userinfo, token/query credentials, quoted credentials in JSON/Python-style mappings, and historical dirty rows.
- Treat `ASHARE_RADAR_LLM_TIMEOUT_SECONDS` as one total budget shared by initial generation, validation, and the optional correction; never reset the outer budget for the correction or add SDK retries.
- Keep primary business functions short and stage-oriented; production Python functions in `app/` and `tools/` are test-guarded at 60 lines and 12 branch points, so split data preparation, rule decisions, persistence, and response assembly before crossing those limits.

### Priority 4: Remaining DataHub Decomposition

`DataHub` already delegates to `QuoteCoordinator`, `KlineCoordinator`, `MetadataCoordinator`, `OrderBookCoordinator`, `DataStatusService`, source-plan helpers, and runtime/status helpers. `ProviderRuntime` owns the shared provider-attempt iterator and timed calls, and `StockPoolResolver` owns the stock-pool/profile cache/provider/fallback state machine. Remaining decomposition work should focus on the wide cache facade and cross-cutting status projections:

- Keep quote/K-line/metadata coordinators responsible only for domain row validation, fallback-cache policy, coverage semantics, and response shaping.
- Split narrow cache protocols such as quote cache, K-line cache, metadata cache, runtime-event sink, and provider-status store for easier fake objects in tests, while keeping `SQLiteCache` as the composition root.
- Move deeper provider status projection policy out of persistence CRUD if provider/capability state grows again.

## 3. Change Checklist

For any functional change:

1. Identify the affected SRS requirement.
2. Update design docs if boundaries or data flow change.
3. Add or update unit tests.
4. Run targeted backend/frontend checks while iterating.
5. Run Ruff, incremental mypy, JavaScript syntax, `pip check`, and the coverage-enabled pytest suite before delivery.
6. Regenerate `docs/FUNCTION_INVENTORY.md` when Python functions/classes move, then run `tools/architecture_inventory.py --check`.
7. Regenerate `docs/API_REFERENCE.md` when routes change, then run `tools/api_inventory.py --check`.
8. Rebuild both Python 3.12 locks after a runtime dependency change, or `requirements-dev-lock.txt` after a development-only change; verify a clean `--require-hashes` install.
9. Smoke test `/api/health` and at least one stock workbench request.

## 5. Invariants

- `standard_symbol` format should remain `CODE.MARKET`, for example `600519.SH`.
- Cache fallback must be visible through source or quality notes.
- Demo provider must remain disabled by default.
- Alert triggers must honor cooldown.
- Recovery events must not spam when state is unchanged.
- Advice history should dedupe repeated same conclusions within the configured window.
- Advice comparison, de-duplication/insertion, and any watchlist unread increment must remain one `BEGIN IMMEDIATE` transaction across cache instances. Only a new, comparable, changed conclusion increments unread; first, unchanged, legacy/version-changed, and de-duplicated snapshots do not. Any unread failure rolls back the insert.
- Mark-viewed must clear only through a positive advice ID that exists for the same symbol, then recompute comparable changed conclusions after that watermark in the same transaction. Missing watermarks preserve unread state so a client cannot erase changes it did not display.
- Empty-query SSE fallback must not turn an all-excluded watchlist back into configured seeds. Seeds apply only when the watchlist table has no rows; explicit symbol queries may still request an excluded symbol.
- Daily cache reuse, cache diagnostics, and K-line quality must use the same 15:15 current-day publication threshold; quote and minute freshness must use the same Shanghai session boundaries.
- Minute analysis must never merge rows across Shanghai-local dates, and `availability=unavailable` must not expose actionable levels or T-plan ranges even when filtered audit `klines` are present.
- Daily range switching is local over the 240-row workbench payload. Advice timeline and minute requests use independent abort/sequence state; a new minute interval costs one request and an unchanged interval costs none.
- A current timeline request must claim the panel with its loading state immediately; symbol plus load sequence must reject every older A-B completion after an A-B-A switch.
- Minute no-content/empty/non-object responses and report symbol/interval mismatches must clear minute state and render unavailable rather than preserve loading or stale chart data.
- Older quote events must never overwrite newer snapshots or same-day history. Compare normalized market epochs, use fetched-at epoch only as the equal-event tie-breaker, and keep legacy UTC ISO rows compatible while persisting new Shanghai fixed-format timestamps.
- Browser mutations and refresh writes must pass both Host-derived-origin and Origin/Referer allowlist checks before side effects, rejecting Host/DNS rebinding; ordinary read-only and non-browser requests remain available.
- Internal model `ValidationError` responses must stay generic `503` and must not expose rejected provider/database values.
- Blocking provider work stays in the bounded runtime-owned executor, SQLite work stays off the event loop, and DataHub shutdown reaches `aclose()` on normal and failed startup paths.
- Provider errors, including quoted secrets in JSON/Python-style mappings, must be sanitized both before persistence/response assembly and after reading historical status rows.
- Scheduler task-run cancellation is terminal: a cancellation may replace a racing success, while late success/failed updates cannot replace `cancelled`.
- Scheduler fallback or incomplete-but-usable refreshes must persist `degraded`, not `success`; a process that does not own the scheduler lock must report live `standby` state and must not be diagnosed as a stopped scheduler while another owner holds the lock.
- Dirty local user rows should stay displayable but must not become executable signals, absorb fresh advice snapshots, or write non-finite values back into alert events.
- A failed local side effect after successful data acquisition must not clear previous good metadata/chart-mark cache or turn a usable workbench response into a global load failure.
- Runtime cleanup must keep historical market tables by partition: `quote_history` per symbol, `kline_minute` per symbol and interval, and `stock_concept` per symbol. `cache_event` and `alert_event` use their own configurable whole-table caps (`ASHARE_RADAR_MAX_CACHE_EVENT_ROWS`, `ASHARE_RADAR_MAX_ALERT_EVENT_ROWS`). Scheduled cleanup uses only regenerable specifications; explicit cleanup previews user-history candidates, creates a verified backup before deleting any, and excludes advice referenced by review plans.
- Import preview claims are single-use and expire; path, mode, exact bundle, and current user-data state must still match at commit. Claim consumption, verified backup creation, and transactional import stay serialized under the shared local-data lock, and backup failure must occur before mutation.
- Alert-event pagination must use event `id` as the authoritative order/cursor. A legacy `after_created_at` value may be accepted only alongside `after_id` and must not filter or reorder events.
- Notification pagination must fully drain or leave the persisted cursor unchanged. Delivery advances the cursor only through the successful prefix; a failed trigger and every later event remain eligible for ordered retry, while a later successful poll clears the visible delivery warning. Disabling must persist the preference, invalidate polling, and clear the cursor; re-enabling must establish a new baseline so disabled-period events are never backfilled.
- Advice-review history must retain evaluations across revisions while current detail projects only the current revision's latest result. Deleting a plan must cascade all of that plan's evaluations without deleting its source advice snapshot, and stale evaluation/history work must not restore deleted UI state. UI-selected review and scan cutoff dates must remain non-future and must never admit later daily bars; custom scans keep the same fixed allowlist and 50-symbol cap as watchlist scans.
- Cache diagnostics must evaluate stock-pool and plate freshness as first-class domains. Non-empty metadata without a usable timestamp remains a missing-freshness condition, while an empty cache remains an explicit availability problem.
- LLM output must pass shape, authoritative-binding, action, conclusion, and numeric-grounding checks before replacing the rule answer. The first validation failure may cause one correction request only, and both attempts share one total timeout; request/timeout failure or a failed correction must use the rule answer. Correction prompts must not echo the previous raw model output or expand the documented remote-data boundary.
- Mobile source/visual/focus order and ARIA states must stay synchronized. Watchlist, alert, and note writes use independent request scopes; only stale UI/readback tails may be cancelled or ignored. Queue metadata changes and mark-viewed actions are local SQLite writes and must not fetch a quote.
- Provider adapters must not turn malformed critical prices into zero-valued quotes or K-lines; Tencent quotes must also keep code/name/timestamp and open/current-vs-high/low containment valid.
- Research reports, rule matches, minute plans, and market-regime outputs must not serialize `nan` or `inf`; invalid feature snapshot fields, forward returns, target K-lines, Alpha verdict context, risk/reward ratios, rule evidence inputs, and intraday zones should stay pending or use bounded conservative fallbacks. Leadership concept evidence and market-regime evidence must drop blank names before stable sorting/text assembly, and doing-T validation must not be worded as a new-position recommendation.
