# 逻辑与架构缺口审计

> 审计对象：2026-07-15 当前工作树（含未提交修改）。本文只记录可由现有调用链触发、且能定位到源码的缺口；不包含泛化功能建议。行号链接以本次审计时的工作树为准。

## 严重度口径

- **P1 / 高**：可让核心行情、研究结论或运行健康状态出现静默错误，正常路径即可触发。
- **P2 / 中**：会造成错误分类、批处理失效或关键反馈丢失，需要特定数据/故障条件触发。
- **P3 / 低**：主要是维护边界或测试盲区，当前影响受限，但容易在后续修改中演化为用户可见错误。

## P1 / 高

### 1. 新拉取的 K 线绕过业务时效校验，陈旧数据会被当作成功结果落库

- **位置**：日线和分钟线只有缓存命中分支调用业务时效判断（[datahub_klines.py:51](../../app/services/datahub_klines.py#L51-L74)、[datahub_klines.py:87](../../app/services/datahub_klines.py#L87-L122)）；供应商返回值仅做结构清洗后即记成功并保存（[datahub_klines.py:145](../../app/services/datahub_klines.py#L145-L157)、[datahub_klines.py:186](../../app/services/datahub_klines.py#L186-L211)）。现成的交易日/交易时段时效规则只在缓存判断中使用（[datahub_cache.py:114](../../app/services/datahub_cache.py#L114-L178)）。
- **触发路径**：缓存未命中、缓存已过期或调用方传入 `use_cache=False` -> 第一优先级供应商返回非空且字段合法、但最后一根日线停留在更早交易日（或分钟线停留在上一交易时段） -> `_fetch_from_priority` 将尝试记为成功、写入当前 `fetched_at` 并直接返回，不再尝试后备供应商。
- **影响**：工作台、研究评分、告警和调度刷新都可能消费陈旧 K 线；供应商健康统计同时显示成功。由于新写入的 `fetched_at` 很新，后续缓存与诊断还会延长这次静默错误的存活时间。
- **修复方向**：在清洗供应商结果之后、记录成功和保存之前，复用统一的业务时效判定；将“结构合法但业务过期”作为可降级的供应商协议错误，使优先级链继续尝试下一供应商。历史区间查询如允许旧尾点，应显式传入不同语义，不能靠跳过校验实现。
- **测试建议**：补充“首供应商返回旧日线/旧分钟线时切换后备源”“所有供应商均旧时不落库且返回聚合错误”“午间、收盘后、周末与节假日边界”“`use_cache=False` 仍校验业务时效”；同时断言供应商成功/失败计数与最终落库内容。

### 2. 系统健康以入库时间替代行情事件时间，且调度器与诊断服务各自定义不同阈值

- **位置**：缓存统计对报价、日线、分钟线都取 `MAX(fetched_at)`（[cache_stats.py:18](../../app/repositories/cache_stats.py#L18-L24)）；调度器据此按自己的报价/K 线阈值判断（[scheduler.py:255](../../app/services/scheduler.py#L255-L273)、[scheduler.py:935](../../app/services/scheduler.py#L935-L964)），系统诊断又维护一套固定阈值（[system_diagnostics.py:42](../../app/services/system_diagnostics.py#L42-L68)、[system_diagnostics.py:218](../../app/services/system_diagnostics.py#L218-L224)）。调度摘要在没有命中上述事件时直接宣称报价、K 线、行业背景和数据源正常（[scheduler.py:880](../../app/services/scheduler.py#L880-L892)），但这里并未对行业背景建立对应时效判断。
- **触发路径**：供应商在当前时间返回旧市场时间的数据 -> 以当前 `fetched_at` 落库 -> 调度监控和系统诊断都判断“新鲜”；反向路径是周末/休市日没有重新抓取，但最后一个有效交易日数据完整 -> 固定墙钟年龄越界并误报陈旧。默认 K 线阈值还会在调度器与诊断之间产生不一致结论。
- **影响**：运维界面会同时出现假绿、假红及跨入口互相矛盾的健康状态；行业背景缺失也可能被笼统的“正常”文案掩盖。该问题会放大上一条陈旧 K 线落库缺陷。
- **修复方向**：缓存统计同时暴露“最近抓取时间”和“最大市场事件时间/业务日期”；由一个共享的、交易日历感知的策略计算各数据域状态，调度器和诊断服务只消费该结果。摘要只能声明实际检查过的数据域，抓取活跃度与市场数据新鲜度应分别呈现。
- **测试建议**：覆盖“新 `fetched_at` + 旧事件时间必须告警”“周末旧 `fetched_at` + 最新有效交易日不误报业务过期”“缺少行业背景不得显示全正常”“相同输入下调度器与系统诊断结论一致”。

### 3. 真实因子报告的汇总校准样本数恒为 0，历史校准无法参与既定风控规则

- **位置**：当前报告固定构造七个因子并总是包含估值锚（[research_factor_current.py:19](../../app/services/research_factor_current.py#L19-L37)）；估值锚的校准对象又固定写为 `sample_count=0`（[research_factor_current.py:162](../../app/services/research_factor_current.py#L162-L190)）。汇总样本数对全部因子直接取最小值（[research_factor_report.py:45](../../app/services/research_factor_report.py#L45-L51)），随后市场状态用该值判断是否允许正向因子降低风险（[research_regime.py:414](../../app/services/research_regime.py#L414-L423)），并在低于 8 时固定输出低样本建议（[research_regime.py:531](../../app/services/research_regime.py#L531-L535)）。
- **触发路径**：任意股票生成因子实验室 -> 六个可回测因子即使各有充足样本，固定零样本的估值锚仍参与 `min(...)` -> `calibration_sample_count` 必为 0 -> 正向因子的样本门槛永远不成立，报告永远附加“样本偏少”。
- **影响**：展示的“最低单因子有效样本”与真正可校准因子的支持度不符；市场状态的正向风险修正被永久禁用，研究结论形成系统性保守偏差。现有测试只用所有因子均有样本的合成列表验证取最小值（[test_research_factor_modules.py:28](../../tests/test_research_factor_modules.py#L28-L44)），没有跑真实七因子组合，因此未发现恒零不变量。
- **修复方向**：明确区分“参与总分”和“参与历史校准”的因子；聚合最小样本数只统计具有可比历史定义且实际完成校准的因子，同时单独披露未校准因子。若业务坚持所有加权因子必须有历史样本才能宣称整体校准，则下游不应再用这个必为零的整体值控制部分可校准因子的风险修正，应改用满足条件的因子子集证据。
- **测试建议**：用 `build_current_factors` 的真实输出构造报告，断言六个已校准因子样本充足而估值锚缺数据时，汇总口径与产品定义一致；覆盖风险修正规则可达、未校准因子仍被披露、所有可校准因子确实无样本时仍返回 0。

## P2 / 中

### 4. 告警批处理只隔离少数异常类型，单条规则的数据库/网络异常可终止整批评估

- **位置**：规则循环只捕获 `RuntimeError`、`TimeoutError` 和 `ValueError`（[alerts.py:34](../../app/services/alerts.py#L34-L56)），但单条评估包含行情访问、质量判定、告警状态持久化及事件回读（[alerts.py:74](../../app/services/alerts.py#L74-L88)）。调度任务依赖 `failed_count` 汇总降级（[scheduler.py:628](../../app/services/scheduler.py#L628-L642)），未捕获异常会绕过该汇总。
- **触发路径**：任意一条规则在仓储层抛出 `sqlite3.Error`，或供应商/连接层抛出未包装的 `OSError`/连接异常 -> 异常逃出 `evaluate_rules` -> 后续规则不再执行，API/调度任务直接失败。现有持久化失败测试只覆盖 `RuntimeError`，没有覆盖真实仓储异常族。
- **影响**：一条损坏规则或一次局部 I/O 故障会扩大为整个告警批次失效；用户拿不到逐条失败结果，调度器也无法区分部分失败和全量失败。
- **修复方向**：在单规则边界捕获可恢复的 `Exception` 并生成脱敏失败结果，同时保留取消信号的传播；仓储/供应商层最好统一包装为领域异常，避免长期维护异常白名单。批次级初始化失败仍应整体抛出。
- **测试建议**：让首条规则分别抛出 `sqlite3.OperationalError`、`OSError` 和未知领域异常，断言后续规则继续执行且 `failed_count` 正确；另测取消异常不会被吞掉、错误信息不泄露连接细节。

### 5. 研究权重画像把“换手率缺失”当作 0，可将高成交额股票误判为大市值稳健股

- **位置**：画像输入先把缺失的成交额、市值和换手率都转换为 `0.0`（[research_factor_weights.py:71](../../app/services/research_factor_weights.py#L71-L78)）；“大市值稳健股”的第二个条件是成交额达到阈值且换手率低于阈值（[research_factor_weights.py:100](../../app/services/research_factor_weights.py#L100-L109)）。
- **触发路径**：`market_cap` 缺失或低于大市值门槛，`amount >= 30 亿元`，但 `turnover_rate is None` -> 缺失值被改写为 0 -> `amount >= 30 亿元 and turnover < 2%` 成立 -> 选择“大市值稳健股”权重模板。
- **影响**：缺数据被错误解释为低换手，进而改变估值、趋势、风险等因子权重和总分；同一股票仅因供应商漏字段就可能切换研究画像，属于错误优先级而非保守降级。
- **修复方向**：保留 `None` 语义；依赖换手率的分支必须要求该字段真实存在。缺关键字段时进入明确的默认/低置信画像，并把缺失信息传到报告，而不是用业务上有效的数值 0 代替。
- **测试建议**：参数化覆盖 `turnover_rate=None/0/1.9/2.0` 与市值门槛两侧；重点断言“高成交额 + 换手率缺失”不会命中稳健画像，同时确认真实 0 换手仍按既定业务规则处理。

### 6. 前端告警评估摘要在同一次成功流程中被立即覆盖，部分失败对用户不可见

- **位置**：手动评估收到结果后先把成功数、触发数和失败数写入 `alertEvents`（[alerts.js:45](../../static/js/alerts.js#L45-L60)、[alerts.js:268](../../static/js/alerts.js#L268-L277)），紧接着 `finishAlertMutation` 重新读取规则和事件（[alerts.js:106](../../static/js/alerts.js#L106-L118)）；事件回读成功后又对同一个 `alertEvents` 容器整体赋值（[alerts.js:121](../../static/js/alerts.js#L121-L137)、[alerts.js:253](../../static/js/alerts.js#L253-L265)）。
- **触发路径**：手动检查返回 `failed_count > 0`（也适用于全成功摘要） -> `renderAlertEvaluation` 短暂渲染“检查部分完成” -> 后续事件列表 GET 成功 -> `renderAlertEvents` 覆盖摘要。网络越快，摘要越接近完全不可见。
- **影响**：用户最终只能看到事件历史，无法确认本轮检查覆盖了多少规则、是否有失败；后端已经实现的逐批降级反馈在正常成功路径被前端丢失。现有测试只直接调用摘要渲染器验证文案（[test_frontend_app_flow.py:1746](../../tests/test_frontend_app_flow.py#L1746-L1764)），没有验证完整 `evaluateAlerts` 流程的最终 DOM。
- **修复方向**：为评估状态设置独立、持久的状态区域，或先完成规则/事件回读再渲染本轮摘要；不要让瞬态操作结果与事件历史竞争同一个 `innerHTML` 所有权。股票切换时仍沿用现有 `isCurrent` 保护清理旧摘要。
- **测试建议**：模拟评估返回部分失败且两次回读均成功，等待整个 `evaluateAlerts` 完成后断言最终界面仍展示失败数；再覆盖事件回读失败、股票切换和连续两轮检查，确保旧摘要不能覆盖新股票/新一轮状态。
